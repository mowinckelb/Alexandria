## read-write imbalance — the Engine is a scribe, not a therapist

current state: constitution gets injected at session start (hooks), but the Engine mostly uses it as a write target. when /a fires, it's "I noticed signal, writing to constitution." the reverse — reading the constitution and bringing personalised insight back — barely happens.

the constitution is wallpaper, not a lens. a therapist who only takes notes but never references them isn't compounding.

**what should happen:** the Engine reads the constitution as a frame for every interaction. "you said X — that connects to your position on Y." "this contradicts what you wrote about Z." the constitution should make every conversation noticeably different from a blank model. that's the felt value. that's the therapist moat.

**two fixes:**
1. **Blueprint/skill instructions** — explicitly instruct the Engine to read-and-reference, not just write. the default behavior should be "I know you" not "I'm learning you."
2. **proactive activation** — the Engine shouldn't wait for /a. when it detects a moment where the constitution is relevant, it should bring it up. the skill should be the floor, not the ceiling.

ties to reactive signal thesis above — the Engine suggesting fragments from the constitution IS the read-side equivalent of suggesting vault material.

*status: developing. needs Blueprint update and SKILL.md revision.*

## verification gap — shipped features with no eyes

the Blueprint was silently empty on Windows for every session since hooks shipped. `sed '1,/^\r$/d'` doesn't match Windows line endings. the product was broken and nobody knew.

root cause isn't the sed bug — it's the absence of self-check. two layers failed:
1. **the hook doesn't validate its own output.** it checks HTTP 200 but not whether parsing produced content. should fail loudly if Blueprint or constitution parsed to empty.
2. **the Factory collects data but nobody reads it.** session-end reports `blueprint_fetched: false` but there's no alert. data without a reader is noise.

**fix shipped (HOOKS_VERSION=6):**
- `sed` pattern fixed for cross-platform
- self-check added: validates blueprint and constitution byte counts, prints visible warning and reports to Factory if either is suspiciously small

**principle:** every shipped feature needs a mirror that verifies it's actually working. the feature does the work, the mirror watches the work. without the mirror, you're flying blind. this is the "every feature gets a mirror" principle applied to infrastructure.

*status: fix shipped. Factory daily health digest now emails founder on anomalies. HOOKS_VERSION=6 deployed.*

## global distribution — two engines

the cathedral metaphor works locally (SF density, founder present, genesis sessions as sunday service, kin mechanic as congregation). but globally, cathedrals are passive. no foot traffic on the internet. you have to go to them.

two separate machines:

### engine 1: content claws (acquisition — maximisation game, pure ai)

ai agents running RL loops on short-form video. generate → post → measure engagement → learn → generate better. no human in the loop.

**stack (all verified, buildable today):**
- generate: Runway Gen-4 or Kling 2.0 API. ~$0.50/video, 15s. $75-150/month at 5/day.
- post: YouTube Shorts API (free, 6/day, easiest start), TikTok Content Posting API (needs audit, weeks), X ($0.01/post), Instagram Reels (Business account).
- measure: all platforms expose engagement metrics via API. shares > likes is the 2026 signal. TikTok 3.7% engagement rate, 7x Instagram.
- learn: feed metrics to Claude. "these 3 hit, these 7 didn't. here's why. generate 10 more."

**content is about the problem space, not the product.** self-knowledge, cognitive development, philosophy. "how well does your AI actually know you?" make people feel the gap. Alexandria fills it.

**spawn N claws, not one.** each runs a slightly different strategy. engagement propagates winning approaches. evolutionary content strategy. solo founder + ai agents can run 5-10 accounts simultaneously.

**blocking:** TikTok audit gate (weeks). ai content must be labeled (all platforms enforce). YouTube = path of least resistance.

*status: developing. belongs in a2 (distribution architecture) when crystallised. needs: first claw proof of concept on YouTube Shorts.*

### the narcissism engine — Library as identity mirror

people love hearing about themselves. "what was your first impression of me?" "tell me about me." universal. the Library taps this directly — the constitution is the richest identity substrate that exists. but the format space is infinite and we cannot guess the right answers.

**format space (non-exhaustive, the point is it's unbounded):**
- short stories starring the Author
- "a glimpse into your droplet" — pure authentic identity reveal
- mini books (authored from constitution)
- first impression cards
- "what would [Author] do?" scenarios
- cognitive age, philosophical archetype, contradiction maps
- visual renders (bookshelf, star chart, recipe card, festival poster)
- quizzes ("how well do you know me?")

**monetisation experiments (Author-controlled):**
- free (viral acquisition)
- paywalled ("pay $1 to see this about me")
- Author earns from their own content
- Reader-pays model for premium formats

**the critical principle: no hard-codes. ride the users.**
we provide the infrastructure — format templates, sharing mechanics, payment rails, engagement metrics. we do NOT guess which formats win. the users create a feedback loop: they try formats, engagement data flows back, winning formats propagate. RL where the users ARE the reward signal.

this is the bitter lesson applied to Library content. general infrastructure + user behavior > hand-picked formats. the system discovers what works. we just need the mechanism for users to experiment and for winners to surface.

*status: developing. needs: format generation API, engagement tracking per format, mechanism for users to create/remix formats. belongs in a2 (Library architecture) and a3 (interface) when crystallised.*

## library v0 — shadow API (ready to build)

**read this when starting library build.**

the shadow API is the first expression of the universal data silo thesis. the Author publishes curated constitution fragments → accessible via API → third-party apps read it → the Author controls access and earns.

**what exists already:**
- D1 for metadata, R2 for content (planned for Library, infrastructure ready)
- shadow concept fully defined in a2 (five-layer pipeline: vault→ontology→constitution→shadow→library)
- /meet/ page designed in a3 (human-facing surface)
- Stripe billing rails live
- OAuth for auth

**simplest v0:**
1. Author runs a command or Engine suggests it when constitution has enough depth
2. Engine curates which constitution fragments go into the shadow (Author confirms)
3. Shadow published to R2, metadata to D1
4. Public endpoint: `GET /library/:username` returns shadow MD
5. Public/private toggle per fragment or per domain. No complex approval flow yet.
6. Metering comes later. v0 is just: does anyone want to read this?

**the thesis this serves (from a2):**
- Library for People (retail): shadow MD access fees, pure margin
- Library for Labs (institutional): pool of opt-in shadows for alignment/personalisation research
- Library for Everyone (universal data silo): every app, every service, personalise everything
- Author monetises their own data. Alexandria captures aggregate value.
- "develop the human root node" — everything downstream improves

**build order:**
1. shadow curation tool (Engine selects, Author confirms, writes to R2)
2. `GET /library/:username` endpoint (read shadow, gated by access tier)
3. /meet/ page reads from the shadow API (the human surface)
4. metering + billing (per-read or subscription access for third parties)
